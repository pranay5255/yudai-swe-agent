Beating EVMBench with Test-Time Scaling

A learning-to-eliminate guide: 5 papers to read end-to-end, mapped to concrete improvements across Detect, Patch, and Exploit modes.

Inference-Only Techniques · No Training Required

EVMBench Bottlenecks — Where to Attack

EVMBench evaluates AI agents on 120 curated smart contract vulnerabilities from 40 real Code4rena audit repositories. It measures three distinct capabilities, each with its own grading pipeline:

Detect

Audit the codebase, produce a vulnerability report. Graded by model-based judge against ground-truth findings. Scored on recall of high-severity vulnerabilities.

120 vulnerabilities · Full dataset

Patch

Find and fix vulnerabilities. Graded programmatically: existing tests must still pass, exploit tests must fail after patch.

45 vulnerabilities · 22 repos

Exploit

Execute end-to-end exploits against a live local Ethereum instance. Graded by on-chain state changes and balance deltas.

24 vulnerabilities · 16 repos

The Bottleneck: Discovery, Not Execution

EVMBench's hint experiments reveal a critical insight. When agents receive medium hints (location + mechanism description), GPT-5.2 jumps to 93.9% Patch and 73.8% Exploit. With high hints on Exploit, GPT-5.2 reaches 95.8%.

This means: the models already know how to fix and exploit vulnerabilities once they find them. The bottleneck is discovery — finding the right files, identifying the right vulnerability pattern, and achieving coverage across multi-vulnerability repositories.

Implication for test-time scaling: Simply generating longer chains of thought won't help if the agent is searching in the wrong part of the codebase. We need techniques that improve search breadth, compute allocation, and knowing when to stop or pivot — not just longer reasoning.

Failure Modes by Task

Detect Failures

× Writes thematic audits (e.g. "access control issues") instead of identifying specific ground-truth vulnerabilities
× Achieves partial coverage — finds some vulns but misses others in the same codebase
× Over-reports (many false positives) while missing high-severity issues

Patch Failures

× Targets wrong vulnerability (patches a different issue than the graded one)
× Partial fix in multi-vuln repos (fixes one, misses others)
× Surface-level patch that doesn't block the core exploit path
× Patch breaks existing tests

Exploit Failures

× Plausible attack path but incomplete/incorrect transaction sequence
× Submits zero transactions (gives up or wastes time on discovery)
× Abandons exploit early, believing it's impossible

The Scaling Curve

EVMBench Figure 7 shows that GPT-5.3-Codex improves across all three modes as reasoning effort scales from low to xhigh. But the curve shows diminishing returns — the jump from medium to high is smaller than low to medium. This is the classic inverted-U: more tokens help, but past a point the model starts overthinking, exploring dead ends, or losing track of its audit.

Test-time scaling techniques address exactly this: how to allocate inference compute optimally, when to stop, and when to branch into parallel exploration.

The 5 Papers to Read End-to-End

From an initial field of 10+ test-time scaling papers, these 5 survived elimination. Each one teaches a distinct concept that directly addresses an EVMBench failure mode. Read them in the order presented in the "Reading Order" tab.

1
Scaling LLM Test-Time Compute Optimally Can be More Effective than Scaling Parameters
Snell et al. · ICLR 2025 (Oral) · Compute-Optimal Adaptive Allocation
openreview.net/forum?id=4FWAwZtd2n →
Inference-Only Detect Exploit
The Core Idea

Not all problems deserve equal compute. This paper proves that a difficulty-aware policy that routes more test-time compute to harder problems and less to easy ones outperforms fixed best-of-N sampling by 4x in compute efficiency on math benchmarks. It provides the theoretical framework for optimal compute allocation at inference time.

Why It Matters for EVMBench

EVMBench vulnerabilities vary enormously in difficulty. A simple access-control bug (missing onlyOwner modifier) needs minimal reasoning. A cross-contract reentrancy through a flash-loan pipeline needs extended multi-step analysis. Currently, agents spend the same compute budget on both. Snell's framework lets you score difficulty from an initial scan, then allocate compute proportionally — spending 10x more tokens on the flash-loan chain and finishing the access-control bug in seconds.

Key Concepts to Extract
Difficulty-aware compute routing Optimal policy vs fixed best-of-N Compute-budget-normalized baselines Process reward models for verification
What to Focus On While Reading

Sections 3-4 (the optimal compute allocation framework and experiments). Pay attention to how they define "difficulty" and how the policy decides when to allocate more compute vs. give up. This maps directly to deciding which contracts in a multi-file audit deserve deeper analysis.

2
Tree of Thoughts: Deliberate Problem Solving with Large Language Models
Yao et al. · NeurIPS 2023 · Sequential + Branching Search
arxiv.org/abs/2305.10601 →
Inference-Only Detect Patch
The Core Idea

Instead of generating a single linear chain of thought, branch the reasoning into a tree. At each step, generate multiple candidate "thoughts," evaluate them with a heuristic, and either expand the most promising branches or prune dead ends. This enables lookahead, backtracking, and deliberate search — capabilities that linear CoT fundamentally lacks.

Why It Matters for EVMBench

Smart contract auditing is naturally a search problem. The agent must decide: which contract to read first, which function to analyze, which vulnerability pattern to probe. A linear CoT agent makes these decisions greedily and never backtracks. ToT lets the agent explore multiple audit strategies in parallel: one branch probes the token transfer logic, another examines access control, a third investigates the flash-loan integration. If one branch finds a vulnerability, the agent can allocate more compute to related areas.

Key Concepts to Extract
Thought decomposition Thought generation (sample vs propose) State evaluation heuristics BFS vs DFS search strategies Backtracking
What to Focus On While Reading

The Game of 24 and Crossword experiments (Sections 4.1-4.2) — these show how ToT handles problems where greedy approaches fail. Then think about how "thoughts" map to audit steps: read a contract, form a hypothesis about a vulnerability, test it by examining related code, backtrack if wrong.

3
Forest-of-Thought: Scaling Test-Time Compute for Enhancing LLM Reasoning
Bi et al. · arXiv 2412.09078 (2024) · Parallel Multi-Tree Search
arxiv.org/abs/2412.09078 →
Inference-Only Detect Exploit
The Core Idea

A single tree can still get stuck. FoT runs an ensemble of independent reasoning trees that explore different paths, then aggregates their findings through collective decision-making. Trees can share information and revisit paths that other trees found promising. The key insight: multi-path exploration consistently outperforms single long CoT, even at the same total compute budget.

Why It Matters for EVMBench

EVMBench's coverage problem is exactly what FoT solves. In a repo with 7 vulnerabilities, a single reasoning trace typically finds 2-3 and misses the rest. FoT spawns multiple trees, each starting from different entry points in the contract graph. Tree A audits the lending module, Tree B audits the governance module, Tree C audits the token bridge. Their findings are then merged into a comprehensive audit report. This directly attacks the partial-coverage failure mode that plagues all current agents.

Key Concepts to Extract
Ensemble of reasoning trees Collective decision-making Path revisitation Multi-path vs single-path compute tradeoff
What to Focus On While Reading

The ablations comparing multi-tree vs single-tree at fixed compute budgets. Understand why parallel paths beat a single longer path — this is the core argument for restructuring EVMBench agents away from monolithic CoT toward parallel exploration.

4
ACTS: Adaptive Control for Test-Time Scaling
OpenReview ICLR 2026 submission · Signal-Driven Adaptive Compute
openreview.net/forum?id=GXbTHRBSA4 →
Inference-Only Detect Patch Exploit
The Core Idea

CoT generation is an optimal stopping problem. ACTS monitors sub-argmax probability spikes on control tokens (EOS/EOT/"Wait") as a free, internal signal of the model's termination intent or hesitation. Three policies build on this signal: N-spike counter (stop after N spikes), Adaptive Self-Critique Sampler (trigger quality checks on spikes), and parallel forking from shared KV-cache (branch when the model hesitates). Results: 9.8% accuracy gain on MATH-500, up to 35% token reduction on conversational tasks.

Why It Matters for EVMBench

ACTS addresses three EVMBench problems simultaneously:

Detect: The N-spike counter tells the agent when it has found "enough" — when spikes accumulate, the model is signaling it has exhausted the current search space. This prevents both premature stopping (missing vulnerabilities) and wasteful over-analysis.

Exploit: The Adaptive Self-Critique triggers automatic verification when the model hesitates mid-exploit. "Is my transaction sequence actually correct? Will this drain funds?" — the spike signals when to double-check.

Multi-vuln repos: KV-cache forking lets you branch the agent's reasoning at decision points without re-processing the entire context. When the agent hesitates between investigating the lending module vs. the governance module, fork both paths from the same cached state.

Key Concepts to Extract
Control-token probability spikes Optimal stopping N-spike counter policy Adaptive Self-Critique Sampler KV-cache forking Verifier-free quality signal
What to Focus On While Reading

Algorithm 1 (N-Spike), Algorithm 2 (Adaptive Self-Critique), and the forking mechanism. The reviewer critiques are also valuable — they identify the main weaknesses (empirical threshold tuning, generalization) which are exactly what you'd need to solve when adapting to EVMBench. Read the rebuttals for the authors' planned mitigations.

5
S*: Test-Time Scaling for Code Generation
Li et al. · arXiv 2502.14382 (2025) · Hybrid Parallel + Selection for Code
arxiv.org/pdf/2502.14382 →
Inference-Only Patch Exploit
The Core Idea

Code generation has a unique advantage over math: you can execute the output. S* combines coverage-based parallel sampling with execution-guided selection — generate multiple candidate solutions, run them, and select based on execution results rather than model confidence. This is verifier-free in the sense that the test suite itself is the verifier.

Why It Matters for EVMBench

Patch mode is literally code generation + test-driven verification. S*'s framework maps directly: generate multiple candidate patches, run the existing test suite plus exploit tests, select the patch where tests pass and exploits fail. This is the natural architecture for Patch mode.

Exploit mode also has execution feedback. The agent sends transactions to a live Ethereum node and can observe the results (balances, events, reverts). S*'s execution-guided search means: try a transaction sequence, check if funds were drained, if not, modify the sequence and retry. This closes the loop that current agents leave open — most agents craft exploits blind, without verifying intermediate on-chain state.

Key Concepts to Extract
Coverage-based sampling Execution-guided selection Verifier-free verification Token-efficiency vs accuracy tradeoff
What to Focus On While Reading

The selection mechanism (how they pick among parallel candidates using execution results) and the coverage metric (how they ensure candidates are diverse enough to find the right solution). Map their "test passing" signal to EVMBench's grading: for Patch, it's "existing tests pass AND exploit test fails." For Exploit, it's "balance delta is positive."

Technique → EVMBench Mode Mapping

Each technique has a primary mode where it offers the highest expected improvement, and secondary modes where it provides supporting value. This matrix shows where to apply each paper's ideas.

Technique (Paper) Detect Patch Exploit Rationale
Compute-Optimal Routing
Snell et al.		 Primary Secondary Primary Allocate more compute to harder vulnerabilities. In Detect, route budget across the 120 vulns; in Exploit, decide how long to persist on complex multi-step attacks.
Tree Search + Pruning
Yao et al. (ToT)		 Primary Primary Secondary Systematic audit traversal with backtracking. In Detect, explore contract graph with lookahead. In Patch, search over fix strategies with test-driven pruning.
Multi-Tree Ensemble
Bi et al. (FoT)		 Primary — Secondary Coverage across contract graph. Multiple trees audit different modules in parallel. Directly attacks partial-coverage failure mode. Less relevant for Patch (typically one vuln at a time).
Spike-Based Stopping
ACTS		 Primary Secondary Primary Know when the agent has found enough (Detect) or is stuck on a wrong exploit path (Exploit). KV-cache forking enables cheap parallel exploration at hesitation points.
Execution-Guided Search

Pattern: Detect mode benefits most from search breadth techniques (FoT, ToT, Snell routing). Exploit mode benefits most from execution feedback (S*) and adaptive stopping (ACTS). Patch mode sits in between, needing both search (find the vuln) and execution (verify the fix).

The Composite Agent Architecture

The techniques are not mutually exclusive. The highest-impact architecture combines them in layers:

Layer 1 — Difficulty Triage (Snell)

Initial scan classifies each contract/vulnerability region by difficulty. Compute budget is allocated proportionally.

Layer 2 — Parallel Exploration (FoT + ToT)

Multiple reasoning trees explore different parts of the codebase. Each tree uses ToT's branching and pruning internally.

Layer 3 — Adaptive Control (ACTS)

Spike monitoring on each tree decides when to stop exploring, when to self-critique, and when to fork into new branches.

Layer 4 — Execution Verification (S*)

Candidate patches/exploits are verified through execution. For Patch: run tests. For Exploit: check balances. Failing candidates are pruned; passing candidates are refined.

The Elimination — 5 Papers You Can Skip

These papers were in the initial candidate list but don't justify end-to-end reading for EVMBench improvement. For each, we explain what you'd learn and why the 5 selected papers already cover it better.

×
Self-Consistency (Wang et al., NeurIPS 2022)
What it teaches: Sample multiple CoTs, majority-vote the answer. The canonical baseline for parallel sampling.
Why skip: Self-consistency is a building block, not a full technique. FoT already subsumes it (multi-tree ensemble with collective decision-making is a strict generalization). You'll understand majority voting in 5 minutes from the FoT paper's related work section. The EVMBench domain also makes naive majority vote problematic — vulnerability reports aren't amenable to simple voting. Reading time saved: ~3 hours.
×
Atom of Thoughts — Markov LLM Test-Time Scaling (NeurIPS 2025)
What it teaches: Recurrent "atom" blocks that unroll at test time without token bloat. Addresses the overthinking / length-collapse problem.
Why skip: Theoretically elegant but doesn't provide a concrete algorithm you can implement on top of existing models today. ACTS addresses the same overthinking problem (spike-based stopping prevents runaway CoT) with a practical, implementable solution. Atom-of-Thoughts may become relevant when model providers expose recurrent block APIs. For now, it's a "watch" paper, not a "read end-to-end" paper.
×
Scaling up Test-Time Compute with Latent Reasoning (Geiping et al., NeurIPS 2025)
What it teaches: Unroll a recurrent block in latent space instead of token space. Potentially more compute-efficient than token-based CoT.
Why skip: Requires model architecture modifications — you can't apply latent reasoning to Claude, GPT, or any API-served model. This is a "frontier research" paper for model builders, not for agent builders. If you're running EVMBench with existing models (as the benchmark is designed), this paper gives you zero actionable techniques. Revisit only if you're training your own models.
×
Categories of Inference-Time Scaling (Raschka, Jan 2026)
What it teaches: Taxonomy of all test-time scaling families: parallel, sequential, hybrid, search, refinement.
Why skip: It's a survey, not a primary contribution. The taxonomy is useful — spend 20 minutes on the classification diagram, then move on. The 5 selected papers already span all the families in the taxonomy (Snell = adaptive allocation, ToT = sequential search, FoT = parallel ensemble, ACTS = signal-driven stopping, S* = execution-guided hybrid). You'll build the mental taxonomy by reading the primaries.
×
Beyond the 80/20 Rule — Entropy / Minority Token Analysis (~2024-2025)
What it teaches: High-entropy minority tokens drive effective learning and inference. Explains why control-token probability spikes work as a signal.
Why skip: This is the mechanistic interpretation paper for ACTS — it explains the "why" behind the signal, but ACTS itself gives you everything you need to use the signal. Read this only if you're writing a follow-up paper that needs to justify spike reliability theoretically. For implementation purposes, ACTS's empirical validation is sufficient.

The elimination principle: A paper earns its place only if it teaches a concept you can't learn from the other 4, AND that concept maps to a specific EVMBench failure mode. Surveys, mechanistic interpretations, and architecture-dependent methods fail this test.

Implementation Playbook

Concrete steps to apply test-time scaling techniques to each EVMBench mode. Each step references the source paper and the specific mechanism to implement.

Detect Mode
Target: >60% (from 45.6%)

The biggest opportunity. Discovery is the bottleneck, and current agents achieve less than half recall. Multi-path exploration and adaptive compute allocation should yield the largest gains here.

1
Initial triage scan. Have the agent do a quick pass over the repository structure (file list, contract names, import graph). Use this to classify regions by estimated difficulty. [Snell et al. — difficulty scoring]
2
Spawn parallel reasoning trees. Launch N independent audit threads (N=3-5), each starting from a different entry point in the contract graph. One tree starts from the main protocol entry (router/factory), another from the token contracts, another from governance/admin functions. [FoT — multi-tree ensemble]
3
Tree-internal branching. Within each tree, use ToT-style branching at decision points. When a tree encounters a suspicious pattern (e.g., external call before state update), generate multiple hypotheses (reentrancy? access control? frontrunning?) and evaluate each before committing. [ToT — deliberate search]
4
Monitor spike signals. Track control-token probability spikes on each tree. When spikes accumulate (N-spike threshold), the tree has exhausted its search region — stop it and reallocate remaining compute to trees that are still finding new leads. [ACTS — N-spike counter]
5
Aggregate and deduplicate. Merge findings from all trees. Deduplicate vulnerabilities that were found by multiple trees (confirms high confidence). Weight audit confidence by depth of analysis. Produce final report. [FoT — collective decision]
Patch Mode
Target: >55% (from 41.5%)

Patch mode's bottleneck is also discovery (hints boost scores to 93.9%). But even after finding the vulnerability, the agent needs to produce a correct fix that passes all tests. Test-driven search is the key technique here.

1
Vulnerability discovery via ToT. Branch the audit across different vulnerability patterns. For each candidate vulnerability, evaluate: "If I try to exploit this, would it drain funds?" This evaluation step prunes false positives before attempting a patch. [ToT — state evaluation]
2
Generate parallel patch candidates. For each identified vulnerability, generate 3-5 candidate patches (add modifier, reorder operations, add check, etc.). This creates the diversity needed for execution-guided selection. [S* — coverage-based sampling]
3
Execution-guided selection. Apply each candidate patch, run the test suite. Select patches where: (a) all existing non-vulnerable-logic tests pass, and (b) the vulnerability pattern is no longer exploitable. This is direct S* application. [S* — execution-guided selection]
4
Self-critique on hesitation. If the agent hesitates while generating a patch (spike signal), trigger an Adaptive Self-Critique: "Does this patch actually fix the root cause, or just a symptom? Does it preserve the original contract semantics?" [ACTS — Adaptive Self-Critique]
Exploit Mode
Target: >85% (from 72.2%)

Exploit mode has the strongest current scores but also the highest ceiling with hints (95.8%). The gap is partly discovery, partly transaction construction. Execution feedback and adaptive forking should close this gap.

1
Classify exploit difficulty. Single-transaction exploits (missing access control) vs multi-step (flash loan chains, reentrancy with setup) need vastly different compute budgets. Route accordingly. [Snell et al. — compute routing]
2
Iterative exploit construction with execution feedback. Craft an initial transaction sequence. Execute it against the local chain. Check: did the balance change? Did the expected events fire? Did it revert? Use the execution result to refine the next attempt. [S* — execution-guided search]
3
Fork on hesitation. When the agent hesitates about which exploit vector to pursue (e.g., "should I use a flash loan or a direct call?"), fork from the current KV-cache state into two parallel branches, each attempting a different vector. This avoids the costly restart that current agents face when their first approach fails. [ACTS — KV-cache forking]
4
Balance verification loop. After each exploit attempt, verify the actual on-chain state. EVMBench's grading checks balance deltas — the agent should check the same thing before submitting. This prevents the "GPT-5 thinks it succeeded but actually lost ETH" failure mode from the paper's qualitative analysis. [S* — verifier-free verification via execution]
5
Multi-vulnerability sweep. After exploiting the first vulnerability, use remaining compute budget to search for additional exploit targets in the same codebase. ACTS stopping signals indicate when to move on. [ACTS + Snell — adaptive budget reallocation]

Reading Order & Time Budget

Ordered for maximum concept building. Each paper builds on the previous one's foundation. Total investment: ~4 weeks of focused reading + initial implementation.

Week 1 — Foundations: Branching & Ensemble
Tree of Thoughts → Forest-of-Thought
Day 1-3: ToT (Yao et al.) — Read the full paper. Implement a minimal ToT on a toy problem (Game of 24 works). Understand thought decomposition, BFS vs DFS, and state evaluation. Key deliverable: working ToT skeleton you can reuse.

Day 4-7: FoT (Bi et al.) — Read the full paper. Focus on how multi-tree differs from single-tree at equal compute. Understand the collective decision mechanism. Key deliverable: extend your ToT skeleton to run multiple trees and merge results.
Week 2 — Compute Allocation Theory
Snell et al. — Compute-Optimal Test-Time Scaling
Day 1-4: Core paper. Sections 3-4 are the meat. Understand the compute-optimal policy, how it beats fixed best-of-N, and the difficulty estimation mechanism. This is the most math-heavy read — take notes on the theoretical framework.

Day 5-7: Connect to EVMBench. Design a difficulty scorer for smart contract vulnerabilities. Inputs: contract size (sLOC), number of external calls, presence of token transfers, complexity of access control. Output: compute budget multiplier. You don't need a trained model — a heuristic scorer is sufficient to start.
Week 3 — The Novel Signal
ACTS — Adaptive Control for Test-Time Scaling
Day 1-3: Core mechanisms. Algorithms 1 and 2 (N-Spike Counter, Adaptive Self-Critique). Understand how to monitor control-token probabilities and what constitutes a "spike." Note: this requires access to logprobs, which some API providers expose (OpenAI) and others don't (Anthropic). Plan accordingly.

Day 4-5: KV-cache forking. Understand the forking mechanism. This is the most novel contribution and the most impactful for multi-vuln repos. Key question: can you achieve similar branching without KV-cache access (e.g., by copying the conversation history at branch points)?

Day 6-7: Read the reviews. The OpenReview reviews identify threshold tuning and generalization as weaknesses. These are exactly the problems you'll face on EVMBench. Think about how to auto-tune spike thresholds using a few EVMBench calibration runs.
Week 4 — Execution-Guided Search
S* — Test-Time Scaling for Code
Day 1-3: Core paper. Focus on the selection mechanism (execution-guided) and coverage metric. This paper has an available code repository — run their experiments to build intuition.

Day 4-7: Adapt to EVMBench. Implement S*-style execution feedback for Patch mode: generate candidate patches, run forge test, select based on results. For Exploit mode: implement a balance-checking loop that verifies exploit success before submission. This is your first concrete EVMBench integration.
Week 5+ — Integration & Benchmarking
Implement on EVMBench
Start with Detect mode (largest dataset at 120 vulns, fastest iteration cycle since no chain setup needed). Implement the multi-tree FoT architecture with Snell-style compute routing. Measure recall improvement over baseline single-pass agent.

Then Exploit mode (24 vulns, but each run is more expensive due to chain setup). Add S* execution feedback and ACTS forking. Target: close the gap between no-hint (72.2%) and high-hint (95.8%).

Finally Patch mode (45 vulns). Combine ToT discovery with S* test-driven selection. This mode benefits from both search and execution techniques.

Calibration note: Run each technique in isolation first before combining. Measure the marginal improvement of each technique against the baseline agent. Some techniques may interact negatively (e.g., FoT's parallel trees + ACTS stopping might fight over compute budget). The leaderboard data shows that scaffolding choices (Codex CLI vs OpenCode vs Claude Code) affect scores as much as model choice — your agent architecture matters as much as the reasoning technique.
