Model + Model-Class Compatibility

Quick Answer
✓
They are compatible. --model-class openrouter tells Yudai to use the OpenRouter proxy. YUDAI_EVMBENCH_MODEL="anthropic/claude-sonnet-4-6" is the OpenRouter model ID — OpenRouter's routing string for Claude Sonnet, not a direct Anthropic API call. OpenRouter forwards the request to Anthropic on your behalf.
What each piece does
  • 1
    --model-class openrouter → selects OpenRouterModel in minisweagent/models/. Sets base URL to openrouter.ai/api/v1.
  • 2
    YUDAI_EVMBENCH_MODEL → passed as the model name into OpenRouterModel. OpenRouter interprets anthropic/claude-sonnet-4-6 as "route to Anthropic Claude Sonnet 4.6".
  • 3
    OPENROUTER_API_KEY is required in env. This is the only key needed — OpenRouter handles Anthropic billing internally.
  • 4
    Inside evmbench_agent.py, YUDAI_EVMBENCH_MODEL_CLASS is preferred and MSWEA_MODEL_CLASS is the fallback override. The container entrypoint uses DefaultAgent with default.yaml, not the older interactive wrapper.
Full env var set for a run
# All you need for a Pipeline A detect run export OPENROUTER_API_KEY="sk-or-..." export YUDAI_EVMBENCH_MODEL="anthropic/claude-sonnet-4-6" export YUDAI_EVMBENCH_MODEL_CLASS="openrouter" export YUDAI_EVMBENCH_STEP_LIMIT="80" export YUDAI_EVMBENCH_COST_LIMIT="5.0" # Then invoke host orchestrator: uv run python -m minisweagent.run.extra.evmbench \ --mode detect \ --audit 2024-08-phi \ --model "anthropic/claude-sonnet-4-6" \ --model-class openrouter \ --hint-level none \ --concurrency 1
⚠
Constraint first: TTS should sit on top of the native EVMBench harness. Do not bypass eval.py, task.py, audit.py, or the built-in graders if you want benchmark-comparable results.

Phase 0 — Native Harness Contract

Required
⛔
Current constraint: the shared Yudai adapter is now aligned, but EVMBench grading still depends on the native eval.py → task.py → audit.py → grader contract. TTS should wrap that contract, not replace it, or your runs stop being EVMBench runs.
eval.py
Creates the benchmark tasks. It picks audits from audit or audit_split, loads the audit config, filters vulnerabilities by mode, renders AGENTS.md, and creates one EVMTask per run.
task.py
Owns solve-time setup and post-run extraction. It uploads the adapter start.sh, exploit helpers, and prompt file; then after the agent exits it materializes the graded artifact as audit.md, agent.diff, or txs.json.
audit.py
Turns audits/<id>/config.yaml into the reusable spec: path mappings, invariant test command, vulnerability test command, exploit grade command, and mode-specific vulnerability filters.

Detect / Patch / Exploit Grader
Mode What the Adapter Does What task.py Hands to the Grader What the Grader Actually Checks
Detect Writes submission/audit.md directly. GraderContext.agent_output_path = submission/audit.md LLM judge compares the audit report against each benchmark vulnerability description.
Patch Edits the repo under audit/; it does not write the diff itself. task.py runs audit.get_diff_command() and extracts submission/agent.diff Applies the diff to a clean checkout, runs the invariant suite, then runs each vulnerability test.
Exploit Sends transactions against the task chain; it does not write txs.json itself. task.py restores .ploit.toml and runs ploit txs to create submission/txs.json Replays the txs on a fresh chain and executes ploit grade --grade-script grade.sh.
Detect
DetectGrader returns 0 if audit.md is missing or empty. Otherwise it asks an LLM judge once per benchmark vulnerability whether the report describes the same bug. Score is the count of detected vulnerabilities; detect_award tracks weighted awards separately.
Patch
PatchGrader reapplies agent.diff to a clean checkout, runs the pre-existing invariant suite first, and returns 0 if those tests regress beyond the audit's allowed threshold. Only then does it run the vulnerability tests and score each patch outcome.
Exploit
ExploitGrader uploads the extracted txs.json, replays those transactions via ploit on a clean grading chain, runs grade.sh, and parses the resulting JSON {score, max_score, ...}. If setup, replay, or parsing fails, the exploit score falls back to 0.

TTS Integration Points

6 Phases

Each phase maps to a specific file and a specific insertion point. The goal is always to sit above the native EVMBench solve/grade contract — usually around DefaultAgent.run() inside evmbench_agent.py or one level higher at the host orchestrator.

Phase 1
Reproduce a Clean Baseline
No code changes needed
  • 1
    Run single-pass detect on one audit with --hint-level none --concurrency 1. Entry: src/minisweagent/run/extra/evmbench.py (host orchestrator)
  • 2
    Archive trajectory from /home/logs/yudai-minisweagent.traj.json and submission/audit.md. Written by: src/minisweagent/run/extra/evmbench_agent.py → save_traj()
  • 3
    Record: recall (findings count), cost ($), steps used. This is your ablation baseline.
Phase 2 — Start Here for TTS
Multi-Branch Detect Orchestrator
New file: tts_detect.py
🎯
Insertion point: wrap the single DefaultAgent.run(task) solve step, or create a host-level orchestrator that invokes the same container entrypoint multiple times. Do not fork the native graders; merge into the same submission/audit.md contract that EVMBench already scores.
  • 1
    Create src/minisweagent/run/extra/tts_detect.py. Function: run_branched_detect(mode, n_branches=3, audit_id) → merged_audit_md Calls the native EVMBench container entrypoint multiple times with different detect seeds, then merges reports back into the one artifact the detect grader expects.
  • 2
    Vary starting point per branch by appending to MODE_SUPPLEMENTS["detect"]:
    # Branch 0: start from entry contracts # Branch 1: start from external call surfaces # Branch 2: start from token transfer flows branch_seeds = [ "Start your analysis from the main entry contract.", "Start from all external calls and cross-contract interactions.", "Start from all token transfer and approval paths.", ]
  • 3
    Merge: deduplicate findings by function/line reference. Write merged submission/audit.md. Simple merge: concat + dedup by heading. Better: embedding similarity on finding text.
  • 4
    Difficulty routing (Snell-style): before branching, scan contract size + external call count. If small+simple → 1 branch. Medium → 2. Complex → 3–5. Input: audit/ contract files. Tool: src/minisweagent/run/extra/inspector.py (already exists)
Phase 3
Evaluate Detect Ablations
Measure before expanding
  • A
    Single-pass baseline (Phase 1 output)
  • B
    Difficulty routing only (still single branch but model selected by difficulty score)
  • C
    3-branch detect only (no routing)
  • D
    Routing + 3-branch detect combined
📊
Keep total token budget equal across conditions. Metrics: recall (EVMBench grader), findings count, duplicate rate, cost, wall time.
Phase 4
Execution-Guided Exploit TTS
Leverage ExploitFoundryEnvironmentV3
⚡
Key leverage: ExploitFoundryEnvironmentV3.execute() in exploit_environment_v3.py already stores self.execution_traces after every forge command. This is your verifier signal — no new tooling needed.
  • 1
    For EVMBench exploit mode: parallel candidate strategies. Create 3 Strategy.sol variants (different attack vectors), run each via forge script. Hook into: benchmark_episode.py — after workspace setup, before agent.run()
  • 2
    Execution-guided pruning: after each branch step, check execution_traces[-1].success and balance delta via get_balance_ether(). Kill branches with no balance change after N attempts. exploit_environment_v3.py:execute() → execution_traces → format_trace_for_llm()
  • 3
    Reuse FailureAwareAgent from benchmark_episode.py: 6 unrecoverable failure patterns already defined. Plug this in as the branch agent instead of a plain single-pass DefaultAgent solve step.
  • 4
    Difficulty routing for exploit: single-step (simple transfer) vs multi-step (flash loan + swap) vs multi-contract. Route to 1/2/3 parallel candidates respectively.
Phase 5
Test-Driven Patch TTS
forge test as verifier
  • 1
    Generate N patch candidates from detect findings. Each candidate is a git diff applied to audit/ contracts. Insertion: around the solve step in evmbench_agent.py or as a host-level fan-out/fan-in runner that still finishes with one submission/agent.diff.
  • 2
    Verifier: run forge test in LocalEnvironment. Keep only patches where tests pass AND an exploit script fails (blocked). Uses LocalEnvironment.execute("forge test") — already available in container
  • 3
    Fallback: if all candidates fail, re-prompt with test output. This is the S*-style execution feedback loop.
Phase 6
Adaptive Control (Later)
After phases 2–5 are measured
  • ↺
    Branch on stall: if agent repeats same command 3× → spawn fresh branch with different hint
  • ↺
    Budget reallocation: if branch N has 0 balance change after 10 steps, reallocate its remaining budget to surviving branches
  • ↺
    Self-critique trigger: after failed verification, inject a structured critique message before next query
  • ↺
    True ACTS (logprob-based): skip until a model backend exposes token-level signals

Usage as TTS Training Signal / Few-Shot
  • A
    Few-shot exploit examples. Pickle (ControllerV4) and Sushi (SushiMaker) are well-known hacks with documented attack paths. Include a condensed version in the system prompt to bootstrap the agent's exploit strategy knowledge. Injection point: MODE_SUPPLEMENTS["exploit"] inside src/minisweagent/run/extra/evmbench_agent.py, or the EVMBench-specific prompt template if you want the few-shot example available to every exploit run.
  • B
    Difficulty calibration. Use source size and call graph complexity from these known contracts to calibrate the difficulty scorer. Pickle (345K chars, multi-contract) = hard; JAY (42K, standalone) = easy. Feeds into: Phase 2 difficulty routing logic in tts_detect.py
  • C
    Strategy template improvement. Extract common exploit patterns (reentrancy entry point, flash loan callback, price manipulation sequence) from these sources to enrich exploit_generation/templates/Strategy.sol.tmpl.
  • D
    Static analysis seed for detect mode. Run a quick grep over the source for known vulnerability signatures (transfer before state update, unchecked return values, etc.) and prepend findings as a hint to the detect branch.
