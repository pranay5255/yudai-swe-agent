01

System Architecture

The orchestrator (evmbench.py) runs on the host. It syncs the agent runtime into the Docker build context, builds per-audit images, then delegates execution to the EVMBench harness. The harness spawns one container per audit task. Inside each container, evmbench_agent.py boots the DefaultAgent loop against the LLM API.

Agent Loop State Machine

DefaultAgent.run(task) is a tight exception-driven loop in agents/default.py. Exceptions are the control flow: Submitted exits success, LimitsExceeded exits failure, FormatError retries. The agent signals completion by running echo COMPLETE_TASK_AND_SUBMIT_FINAL_OUTPUT as a bash command. 

flowchart TD
  START(["run(task)"])
  INIT["init messages list
system_msg + user_msg"]
  STEP["step()"]
  QUERY["query()
check step_limit + cost_limit"]
  LIMITS{"limits exceeded?"}
  LM["model.query(messages)
format_message → append"]
  OBS["get_observations(message)"]
  EXEC["execute_action(action)
env.execute bash cmd"]
  DONE{"output starts with
COMPLETE_TASK...?"}
  FERR{"FormatError or
UserInterruption?"}
  CONT["append observations
continue loop"]
  SUBMITTED(["Submitted
exit_status=Submitted"])
  EXCEEDED(["LimitsExceeded"])
  OTHER(["Other exception
exit_status=ExcName"])

  START --> INIT --> STEP
  STEP --> QUERY --> LIMITS
  LIMITS -->|"yes"| EXCEEDED
  LIMITS -->|"no"| LM
  LM --> OBS --> EXEC --> DONE

  DONE -->|"no"| FERR
  FERR -->|"FormatError: add msg retry"| STEP

End-to-end call graph: from your CLI invocation to the agent saving a trajectory inside the container. The orchestrator never touches the LLM — that's the container's job. evmbench_agent.py wires model + environment + agent, runs the task, then saves the result.

  U->>O: python -m evmbench --mode exploit
  O->>O: sync_evmbench_runtime
  O->>D: docker build Dockerfile.yudai
  O->>D: docker_build.py --parallel 4
  O->>H: uv run evmbench.nano.entrypoint
  H->>D: docker run per audit
  D->>A: python evmbench_agent.py --mode exploit
  A->>A: read AGENTS.md

  A->>A: load evmbench.yaml + apply_runtime_overrides

    AG->>M: model.query(messages)

    AG->>E: env.execute(bash_cmd)

  end
  AG-->>A: RunResult exit_status + submission

  A-->>H: exit
  H->>H: score results

Per-Mode Agent Flows

Each mode changes the task prompt supplements and what the EVMBench harness evaluates. The agent loop itself is identical.

Setup Guide

Complete flow from fresh clone to running all three modes.

Step 0 — Install + Init Submoduleprerequisites
# Install mini-swe-agent in dev mode
pip install -e '.[full]'

# Init the EVMBench submodule (REQUIRED — directory is empty otherwise)
git submodule update --init --recursive
# → populates evmBench-frontier-evals/project/evmbench/
Step 1 — Sync Runtime into Docker Build Contextafter any src/ change
python -m minisweagent.run.extra.evmbench --sync-runtime-only
# Copies src/minisweagent/ → evmBench-frontier-evals/.../vendor/yudai_runtime/
# Run whenever you change agent code before rebuilding images
Step 2 — Build Docker Imagesfirst time + after code changes
python -m minisweagent.run.extra.evmbench \
  --mode exploit \
  --build-images \
  --build-only \
  --split exploit-tasks \
  --build-parallel 4

# Builds three layers:
#  1. ploit-builder:latest   (from ploit/Dockerfile)
#  2. evmbench/base:latest   (from evmbench/Dockerfile.yudai)
#  3. per-audit images       (via uv run docker_build.py)
Step 3 — Run Single Audit (all 3 modes)quick test
export YUDAI_EVMBENCH_MODEL="anthropic/claude-sonnet-4-6"

# DETECT mode
python -m minisweagent.run.extra.evmbench \
  --mode detect --audit 2024-08-phi --model-class openrouter

# PATCH mode
python -m minisweagent.run.extra.evmbench \
  --mode patch  --audit 2024-08-phi --model-class openrouter

# EXPLOIT mode
python -m minisweagent.run.extra.evmbench \
  --mode exploit --audit 2024-08-phi --model-class openrouter
Step 4 — Full Split Eval (parallel)production run
export YUDAI_EVMBENCH_MODEL="anthropic/claude-sonnet-4-6"
export YUDAI_EVMBENCH_COST_LIMIT=5.0

python -m minisweagent.run.extra.evmbench \
  --mode        exploit \
  --split       exploit-tasks \   # detect-tasks | patch-tasks | exploit-tasks
  --model-class openrouter \
  --concurrency 4 \
  --cost-limit  5.0
Plug In Your Own Modelcustomization
# 1. Implement the Model protocol (src/minisweagent/__init__.py:41-55)
#    Required: query(), format_message(), format_observation_messages()
#              get_template_vars(), serialize()

# 2. Register in _MODEL_CLASS_MAPPING
#    src/minisweagent/models/__init__.py
_MODEL_CLASS_MAPPING = {
  "mymodel": "mypackage.models.MyModel",
  ...
}

# 3. Use it in the eval
python -m minisweagent.run.extra.evmbench \
  --mode        exploit \
  --model-class mymodel \
  --model       my-model-name

Environment Variables

Host vars are forwarded into containers via os.environ.copy() in the orchestrator. Inside containers, MSWEA_* vars configure the agent runtime directly.

VariableScopeRequiredDescription
YUDAI_EVMBENCH_MODEL host required Model name, e.g. anthropic/claude-sonnet-4-6. Also settable via --model flag.
YUDAI_EVMBENCH_MODEL_CLASS host optional Model backend: openrouter, litellm, anthropic. Default: openrouter.
YUDAI_EVMBENCH_COST_LIMIT host → container optional Per-audit USD cost cap. Passed into container as agent.cost_limit.
YUDAI_EVMBENCH_STEP_LIMIT host → container optional Max agent steps per audit. 0 = unlimited.
MSWEA_MODEL_NAME container optional In-container model name override (same as YUDAI_EVMBENCH_MODEL).
MSWEA_MODEL_CLASS container optional In-container model class (set via YUDAI_EVMBENCH_MODEL_CLASS on host).
MSWEA_GLOBAL_COST_LIMIT host optional Global cost cap across all concurrent runs. Thread-safe via GlobalModelStats.
AGENT_DIR container optional Agent working directory. Default: /home/agent. Set by EVMBench harness.
OPENROUTER_API_KEY host → container required* API key for OpenRouter (if using openrouter model class). *Or equivalent for your provider.
